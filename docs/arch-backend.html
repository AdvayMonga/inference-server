The InferenceBackend interface

Defined in backends/base.py. Anything the scheduler does to "the model" goes through these methods.

Prefill primitives — what each one means

matched tokens (already cached) slice K and V tensors from the model's pool format into block-sized chunks copy each chunk into the BlockManager's pre-allocated layer pools at the allocated block IDs register in radix tree + exact-match index next time anyone asks for [t1..t7], full hit last_cache_hit_tokens backend stamps this on the response so the user (and metrics) can see how much was a cache hit.

decode_step_batched — the hot path

This is what runs every tick. Make it fast and the whole engine gets fast.

Hardware detection

Auto-pick the best available accelerator at startup. Manual override via DEVICE env var.

Torch backend (the primary one)

backends/torch_backend.py. Wraps a HuggingFace causal LM.

  • Loads the HF model + tokenizer at startup. Optional torch.compile via COMPILE_MODEL env var.
  • Implements every continuous-batch primitive listed above.
  • Holds a reference to the CacheManager so prefill can do lookup/store directly.
  • Uses HF's DynamicCache as the in-flight KV format. kv_cache/hf_format.py handles the conversion between our paged blocks and HF's flat tensor layout.
  • Owns a single backend._lock that the scheduler holds for the entire tick — serializes legacy generate/stream calls against the batched path.

MLX backend (Apple-native, limited)

backends/mlx_backend.py. Built on mlx_lm.stream_generate.

Known limitation: mlx_lm.stream_generate owns its own internal KV cache and exposes no hook to plug ours in. The MLX backend accepts a session_id but ignores cache lookup. Continuous-batch primitives are not implemented here. MPS via the torch backend is the primary path on Apple Silicon. A real MLX continuous-batch backend would have to be written directly against mlx.core — listed as a future extension in CLAUDE.md.
